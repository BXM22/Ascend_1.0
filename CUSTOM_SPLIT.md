# Custom split & programs

Reference for **workout split types**, **user-created programs**, and how they connect to templates, generation, and the active program. For template generation rules, see [GENERATE_TEMPLATE.md](./GENERATE_TEMPLATE.md). For the Templates tab shell, see [TEMPLATE_VIEW.md](./TEMPLATE_VIEW.md).

---

## 1. Role in the app

Users build **programs** whose **`category` is `.split`** (`WorkoutProgram`). Each program stores a **`WorkoutSplitType?`** (`splitType`) and a **`[WorkoutDay]`** list (one entry per slot in the weekly blueprint—training days and rest).

The **Templates → Programs** area lists these programs via **`WorkoutSplitsSection`**. Creation happens in **`CreateWorkoutProgramView`** (“New program” sheet): name, optional description, **split type** (or **custom day names**), optional **auto-generate** workouts from **`TemplatesViewModel.generationSettings`**, and optional **set as active**.

---

## 2. Data: `WorkoutSplitType`

**File:** [`Ascend/Models/WorkoutSplit.swift`](Ascend/Models/WorkoutSplit.swift)

| Case | Summary | `numberOfWorkoutDays` |
|------|---------|------------------------|
| `pushPullLegs` | 6 training slots / week pattern (PPL ×2 + rest in `dayNames`) | 6 |
| `chestBackLegsShouldersArms` | 3-day pattern repeated | 6 |
| `backBicepsChestTricepsLegsShoulders` | 4 distinct training days | 4 |
| `chestBackLegsShouldersArms4Day` | Chest / Back / Legs / Shoulders & Arms | 4 |
| `upperLower` | Upper / Lower ×2 | 4 |
| `fullBody` | Full-body variants + rest pattern | 3 |
| `custom` | User supplies **`customDays`** strings; `numberOfWorkoutDays` is 0 at type level—length comes from names | — |

**`dayNames`** is a **7-slot week-oriented list** (indices 0…6) for preset splits—see the enum for exact strings (e.g. PPL includes `"Rest"`). **`custom`** uses **`customDays`** in **`WorkoutSplit`** init (see §5); for **`WorkoutProgram`**, custom days come from **`createCustomProgram(dayNames:)`**.

**`description`** is human-readable copy shown on program cards when `splitType` is set.

## 3. Programs: `WorkoutProgram` + `WorkoutProgramViewModel`

**File:** [`Ascend/Models/WorkoutProgram.swift`](Ascend/Models/WorkoutProgram.swift), [`Ascend/ViewModels/WorkoutProgramViewModel.swift`](Ascend/ViewModels/WorkoutProgramViewModel.swift)

- **`WorkoutProgram`** holds `days: [WorkoutDay]`, optional **`splitType`**, **`frequency`**, and **`category`**. Split-style user programs use **`category: .split`**.
- **`createProgram(name:description:splitType:frequency:)`** maps **`splitType.dayNames`** to **`WorkoutDay`** rows (`isRestDay` when name is `"Rest"`). Default description falls back to **`splitType.description`** if the user leaves description empty.
- **`createCustomProgram(name:description:dayNames:)`** builds days from the string list; rest is detected if the name **contains `"rest"`** (case-insensitive). **`splitType`** is **`.custom`**.
- **Templates** attach per day via **`templateId`** on **`WorkoutDay`**; **`assignTemplate` / `removeTemplate`** update **`programs`** and can sync **estimated duration** from **`TemplatesViewModel`**.
- **Active program** state (which program drives “today”, completions, etc.) lives on **`WorkoutProgramViewModel`** (`setActiveProgram`, `clearActiveProgram`, day completion helpers).

**Generation helpers** (same view model) include **`ensureTemplateForDay`**, **`populateProgramDaysWithAutoGeneratedWorkouts`**, and related logic— they take **`WorkoutGenerationSettings`** from **`templatesViewModel.generationSettings`**.

## 4. UI surfaces

| Surface | View | Purpose |
|---------|------|---------|
| Program list | **`WorkoutSplitsSection`** | Lists **`programViewModel.programs`** where **`category == .split`**. Empty state → create. Header **+** opens **`CreateWorkoutProgramView`**. |
| Create / custom split | **`CreateWorkoutProgramView`** (in [`WorkoutSplitsSection.swift`](Ascend/Views/Components/WorkoutSplitsSection.swift)) | Kinetic chrome: name, blueprint, **`ForEach(WorkoutSplitType.allCases)`** cards, **custom day editor** when **`selectedSplitType == .custom`**, **auto-generate** toggle, **set active**, link to **`WorkoutGenerationSettingsView`** via **`templatesViewModel.generationSettings`**. |
| Program card | **`WorkoutProgramCard`** | Expandable days, **Set Active** / **Active**, template pickers, **`DayTypeInfoCard`**, delete. |
| Entry | **`TemplatesView`** | Embeds **`WorkoutSplitsSection`** under **Programs** with **`WorkoutSplitsSection`**. |

## 5. `WorkoutSplit` model (struct) vs programs

The codebase also defines **`WorkoutSplit`**, **`WorkoutSplitDay`**, and **`WorkoutSplitViewModel`** ([`WorkoutSplit.swift`](Ascend/Models/WorkoutSplit.swift), [`WorkoutSplitViewModel.swift`](Ascend/ViewModels/WorkoutSplitViewModel.swift)). That API mirrors split metadata + **`startDate`** / **`getCurrentDayIndex()`**. The **primary user path** for the Templates **Programs** UX is **`WorkoutProgram`** + **`WorkoutProgramViewModel`**; the standalone **`WorkoutSplitViewModel`** is not wired through the same UI as of this doc—treat it as **secondary / legacy** unless you integrate it.
